# Database Schema & Encryption Mapping

## Tables and Encrypted Columns

### 1. `USERS`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `username` | No | – | Primary key, used in joins, must be plaintext |
| `password` | No (hashed) | – | Password hashed with salt (SHA‑256) |
| `store_region` | No | – | Used for price calculation, not sensitive |
| `card_info` | No (not used) | – | Payment gateway integration disabled |
| `company_name` | No | – | Public info |
| `publisher_name` | No | – | Public info |
| `user_type`, `account_status` | No | – | RBAC logic |
| **`encrypted_email`** | Yes | **User’s RSA public** | Personal info – only user (or server) can decrypt |
| `email_sig` | N/A | **User’s ECC private** | Integrity of email |
| **`encrypted_buyer_address`** | Yes | **User’s RSA public** | Personal address – per‑user encryption |
| `address_sig` | N/A | **User’s ECC private** | Integrity of address |
| `rsa_public`, `ecc_public` | No (public) | – | Public keys stored plain |
| `rsa_private_encrypted`, `ecc_private_encrypted` | No (plain JSON) | – | Due to RSA size limit; stored in plaintext but DB file is encrypted at rest |

### 2. `WALLET_BALANCE`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `username` | No | – | Foreign key |
| **`encrypted_balance`** | Yes | **User’s RSA public** | Financial data – only user should see |
| `balance_sig` | N/A | **User’s ECC private** | Integrity of balance |

### 3. `GAME_PUBLISH_REQUEST`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `request_id`, `username`, `game_name`, `game_genre`, `estimated_release_year`, `status`, `payment_status` | No | – | Not sensitive or used for admin approval |
| **`encrypted_basic_description`** | Yes | **Admin RSA public** | Only admin should read the description; no user needs it |
| `desc_sig` | N/A | **Admin ECC private** | Integrity |

### 4. `GAME_LIST`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| All fields except `encrypted_revenue_generated` | No | – | Game metadata (name, genre, price, etc.) are public |
| **`encrypted_revenue_generated`** | Yes | **Developer’s RSA public** | Only the developer who owns the game should see its revenue |
| `revenue_sig` | N/A | **Developer’s ECC private** | Integrity |

### 5. `OWNED_GAMES`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `username`, `game_name`, `purchase_type`, `posted_review` | No | – | Foreign keys and status |
| **`encrypted_amount_paid`** | Yes | **Buyer’s RSA public** | Purchase amount – only the buyer should see |
| `amount_sig` | N/A | **Buyer’s ECC private** | Integrity |

### 6. `WALLET_CODE`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `wallet_key` (the actual secret) | **No** (replaced) | – | We do not store plain wallet keys. Instead we store: |
| **`encrypted_wallet_key`** | Yes | **Admin RSA public** | Generated by admin, redeemable by any user – only server decrypts to validate |
| `key_sig` | N/A | **Admin ECC private** | Integrity |
| `amount`, `status` | No | – | Plain metadata |

### 7. `GAME_KEY`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `game_key` (plain secret) | **No** (replaced) | – | Replaced by: |
| **`encrypted_game_key`** | Yes | **Admin RSA public** | Generated by developer, redeemed by buyer – only server decrypts |
| `key_sig` | N/A | **Admin ECC private** | Integrity |
| `game_name`, `status` | No | – | Plain metadata |

### 8. `Reviews`
| Column | Encrypted? | Key Used | Reason |
|--------|------------|----------|--------|
| `game_name`, `username`, `rating` | No | – | Not sensitive |
| **`encrypted_review`** | Yes | **Admin RSA public** | Review text could contain PII – only server decrypts for display |
| `review_sig` | N/A | **Admin ECC private** | Integrity |

### 9. Other Tables (`WISHLIST`, `CART_SYSTEM`, `SENT_FRIEND_REQUEST`, `FRIENDS`, `otp_codes`)
- No encryption – they contain only usernames, game names, flags, and OTP codes (which are ephemeral and not sensitive).

---

## Why This Mapping?

- **User‑specific sensitive data** (email, address, balance, amount paid, revenue) is encrypted with the **user’s own RSA public key**. This ensures that even if the database is stolen, an attacker cannot read another user’s data without that user’s private key (and the private keys are stored in the same DB, but that’s a separate concern – we rely on file‑level encryption for the DB itself).
- **Server‑only data** (wallet codes, game keys, reviews, publishing descriptions) is encrypted with the **global admin RSA key**. Only the server (holding the admin private key) can decrypt them.
- **No symmetric encryption** – strictly RSA + ECC, meeting the “exclusively asymmetric” requirement.
- **Integrity** for every encrypted field is provided by ECDSA signatures (using the same key pair type as the encryption key: user ECC for user data, admin ECC for admin data).
